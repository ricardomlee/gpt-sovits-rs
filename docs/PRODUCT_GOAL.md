# 产品目标：本地个人语音服务

这个项目不是做一个“Rust 版 GPT-SoVITS demo”。它应该是一个可以长期放在 PC 或 NAS 上跑的本地 TTS 服务。

理想状态很简单：部署好服务，选一个音色，接到自己的助手里。之后助手只发文字和音色名，服务返回语音。普通使用时，不需要我每句话手动调 temperature、top_p 之类的参数。

## 使用时应该是什么感觉

- 部署别折腾：下载二进制或启动 Docker Compose，就能跑起来。
- 数据清楚：模型、音色、输出文件都放在固定目录里，升级不会覆盖。
- 使用简单：日常只选择“音色”，不用关心参考音频路径、prompt 文本和采样参数。
- 接入简单：助手和 agent 框架通过稳定的 HTTP API 调用，不需要了解内部实现。
- 长文本稳定：一段较长回复也能自然分句、合成和拼接。
- 声音可信：尽量减少静音、爆音、截断、节奏突兀和明显机械感。
- 本地私有：正常推理不依赖云服务。

## 这个仓库负责什么

- GPT-SoVITS 本地推理。
- PC / NAS 上的部署。
- 音色配置、HTTP 服务、agent 接入、批量测试和质量检查。
- 速度、稳定性和音质优化。

## 暂时不做什么

- 不在 Rust 版本里做训练和微调。
- 不把每个底层参数都暴露给普通用户。
- 不追求一次性支持所有前端、插件和平台。

训练、微调、制作高质量音色，仍然交给 Python 版 GPT-SoVITS。这个仓库只处理训练好以后怎么用。

## 近期方向

1. 让音色管理更简单：用 `voices/<name>/voice.json` 管理音色。
2. 让助手接入更简单：HTTP API 围绕 `voice + text` 设计，并提供 OpenAI-compatible 入口。
3. 让长文本更稳定：自动分句、拼接、检测截断和坏音频。
4. 让质量验证更自动：少靠手动听，多靠 smoke test 和音频指标先筛问题。
5. 让推理更快：持续优化 CPU / GPU 后端、缓存、解码和 benchmark。

这个项目做得好不好，不看内部堆了多少技巧。看用户能不能轻松部署、轻松接入，并且愿意真的让自己的助手每天用它说话。
